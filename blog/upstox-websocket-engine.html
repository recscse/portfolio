Latency Kills: Building a High-Speed Market Data Engine with Upstox V3

Python WebSockets Protobuf FinTech

In algorithmic trading, data is the new oil, but latency is the enemy. If your execution engine is even 500 milliseconds behind a price move, your edge is already gone.

This post deep-dives into how I architected a production-grade market data engine using Upstox V3 WebSockets and Protocol Buffers (Protobuf) to achieve near-zero lag streaming.

1. From Polling to Streaming

Most basic trading bots use polling—requesting prices via REST APIs every second. This is inefficient and slow. My platform utilizes a persistent WebSocket stream where data flows unidirectionally from exchange servers in real time.

2. Why Binary Protobuf?

Unlike many brokers that send data in JSON text, Upstox V3 uses Protocol Buffers. While JSON is readable, it carries significant network overhead due to text serialization and larger payload sizes.

Feature JSON (Traditional) Protobuf (Optimized)
Data Type Text / String Compressed Binary
Parsing Slow (CPU Intensive) Ultra-Fast (Native Mapping)
Payload Size Heavy Minimal (Up to 80% smaller)

3. Implementation Workflow

Authorization Handshake

Before connecting, we must request a short-lived authorized redirect URI to ensure the WebSocket connection is authenticated and secure. 


def get_authorized_url(access_token):
    url = "https://api.upstox.com/v3/feed/market-data-feed/authorize"
    headers = {"Authorization": f"Bearer {access_token}"}

    return response.json()['data']['authorized_redirect_uri']

Binary Decoding with Protobuf

When binary packets arrive, they must be decoded using the pre-defined Protobuf schema. We map raw bytes directly to structured Python objects. 


import MarketDataFeed_pb2 as pb
from google.protobuf.json_format import MessageToDict

def on_message(raw_binary):
    feed_response = pb.FeedResponse()
    feed_response.ParseFromString(raw_binary)
    return MessageToDict(feed_response)

4. Production Fail-Safes

A real-world trading engine must be resilient. I implemented several critical patterns to guarantee high availability:

  • Exponential Backoff: Automatic progressive reconnect delays to avoid server rate-limiting.
  • Heartbeat Monitoring: If no tick data arrives for 60 seconds during market hours, the engine triggers a silent reconnect.
  • Dynamic Subscription: Allows adding or removing tickers on the fly without interrupting ongoing data streaming.

5. Final Results

By implementing this binary-first architecture, the system processes over 1,000 price updates per second with internal latency under 5 milliseconds.
